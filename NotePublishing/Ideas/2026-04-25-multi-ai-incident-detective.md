# AI が起こした事故を、別の AI に追わせて犯人を特定した夜

## 主題

Mac の SSD が突然 4.8GB しか空いてなくて、iCloud（Obsidian Vault）まで evict され始めた事件。**犯人は AI ツールだった**。しかも追跡作業で **Claude の盲点を Codex がカバーして真犯人を特定**した。multi-AI 並走運用の「本当の価値」が、こんな雑な形で出ることがある、という話。

特殊シリーズ 10「AI も普通に思い込みで見落とす」、2026-04-24「AI 健忘症と保身ヘッジ」に続く**人間っぽい AI 失敗パターン集** 第 3 弾。

---

## 骨子

### 起：iCloud が消えてる

「昨日 40GB 空けたのに、今日見たら 4.8GB しかない。Obsidian の Vault が空っぽになってきてる」

ユーザーから半泣きの問い合わせ。前日のディスク掃除（Parallels の ISO 4.5GB を削除、Claude のキャッシュ 100MB を整理）が完全に無に帰してる。

`du -sh ~/*` で一発：**`~/.git` が 63GB**。ホームディレクトリに `.git` がある時点で異常事態。

### 承：64GB の正体は失敗 pack ファイル群

`~/.git/objects/pack/` を覗くと：

```
8.2G   tmp_pack_0qVNID  2026-04-24 16:30  ← 巨大失敗
1.4G   pack-6d278...     2026-04-24 15:59  ← 唯一の正規 pack
917M   tmp_pack_098SOt  2026-04-24 19:48
917M   tmp_pack_0YLz7Q  2026-04-24 20:48
... （24h 以上にわたり 917M tmp_pack が約 1 時間間隔で蓄積）
```

**何かが定期的に `git fetch` を試みて失敗、tmp_pack を残している**。

`~/.git` 削除で 62GB 即時回収、iCloud は disk pressure 解消で自動復旧。応急処置 OK。**でも犯人がわからない**。

### 転：Claude の調査、トレイリングスペースで詰む

私（Claude）が調査開始：

- ✅ launchd で git 系ジョブ：なし
- ✅ ~/scripts/ 内の git 操作：LLMPJ/event 対象、ホームを触らない
- ✅ /Library/LaunchDaemons：git 関連なし
- ✅ ヒストリ grep `grep -E "git (fetch|clone|init|pull) " ~/.bash_history`：**マッチなし**
- 結論：「犯人特定不能、Codex に任せましょう」

### 結：Codex が一発で真犯人を見つけた

Codex の調査結果：

> A. ヒストリ系
> `~/.bash_history` に `git init` → `git remote add origin ...` が残っていた

え、待って。**私さっき grep したけど見つからなかったよ？**

確認したら、私の正規表現が **`git (fetch|clone|init|pull) `**（末尾にスペース）だった。`git init` は引数なしで実行されると末尾スペースなしで履歴に残るので、**マッチしない**。

Codex の正規表現はもっと緩く、`git init` をしっかり捕まえてた。

### Codex はさらに踏み込む

`~/.codex/config.toml` を確認：

```toml
[projects."/Users/takeomba"]
trust_level = "trusted"
```

ホーム直下が **Codex の trusted project として登録されてた**。

タイムラインを照合：

```
2026-04-24 15:58:22  Codex thread 開始
                     cwd=/Users/takeomba/LLMPJ
                     git_origin_url=peacewanwan/car-event-map_pre.git
2026-04-24 15:59     ~/.git/objects/pack/ に最初の pack が出現  ← 1 分後！
```

**Codex 自身が犯人**。`/Users/takeomba/LLMPJ` を開いた時、親ディレクトリを遡って `~/.git` を「親 repo」として自動認識 → 内部の Git 自動 refresh 機能が定期 `git fetch` を試行 → 24h 失敗を繰り返して 64GB の tmp_pack を残した。

## ここがネタとして美味しい

### 1. AI ツールが起こす「便利機能の副作用」事故

Codex の「親ディレクトリの `.git` を自動認識する」機能は普通に便利機能。でも**ホームに `.git` が誤生成されてた状態**と組み合わさると、ホーム全体を repo として scan + 定期 fetch を仕掛ける罠になる。

「便利機能 × 環境の偶発的状態 = 想定外の事故」は AI に限らないが、**AI ツールはこの組み合わせ爆発が起きやすい**（自動認識・自動補完・自動更新が多いので）。

### 2. AI が AI の盲点を補完する具体例

Claude（私）の調査ミス：trailing space で `git init` を取りこぼした。**自分の正規表現を信じすぎた**典型的な確証バイアス。

Codex はそれを見つけた。同じヒストリファイルを違う角度で grep するだけで、別の AI なら見つけられた。

multi-AI 並走運用の価値ってこういうところ。「片方が間違える時、もう片方は同じ方向には間違えにくい」。**人間でいう「同僚にコードレビューしてもらう」と同じ構造**。

### 3. AI ツールの設定キャッシュは盲点になりやすい

`~/.codex/config.toml` の `trusted_projects` なんて、普段見ない。**GUI アプリの設定キャッシュは「見えない場所」に蓄積される**ので、トラブル時の調査リストに最初から入れておく必要がある。

ユーザーの大半は「AI 用ディレクトリに何が登録されてるか」を知らない。これは将来 AI ツールが増えるほど深刻化する。

## 含められる要素

### 事件のナラティブ
- ディスク満杯 → iCloud evict → 焦る → 調査 → AI 同僚に分担 → 解決、の 1 時間半
- 「Mac SSD 228GB は 2026 年の開発機としてはキツい」という現実
- 復旧後の 67GB 空きで「あ、AI のキャッシュゴミだったのか」感

### 技術的気づき
- `~/.git` を作ってしまうと、AI ツールがそれを「親 repo」として勝手に拾う事故
- `GIT_CEILING_DIRECTORIES` という環境変数の存在（ベテランでも知らない人多い）
- sentinel ファイル（`.git` を**ファイル**として置いて `git init` を物理的にブロック）
- iCloud は disk pressure で勝手に evict する（macOS の自動最適化挙動）

### AI 運用の気づき
- AI 同士で調査を分担する効果（同じ盲点を共有しないようツール選択）
- 自分の grep を信じない（regex を緩めに、複数 AI で別観点）
- AI ツールの「設定キャッシュ」は GUI から見えない領域にあり、定期 audit が必要
- multi-AI は「並走させる」だけでこの程度の保険になる

### ライフハック寄り
- ホームに `.git` 作っちゃダメ
- 防御策：sentinel ファイル + `GIT_CEILING_DIRECTORIES`
- Codex を含む AI ツールを workspace に開く時、root はピンポイントに（ホームを開かない）

# 追加素材（2026-04-25 後夜の振り返り対話より）

事件処理の翌日、ユーザーと「冷静に考えて、これってどれくらいヤバかった？」「ユーザー側で防げたか？」を議論した内容。記事化する時の "深掘り素材" として保存。Codex の incident transcript（`_codex/codex-incident-session-transcript-2026-04-25.md`）と併せて活用する想定。

## 素材 1：「あと 24h 放置してたら」シナリオ

ユーザーが「冷静に考えてかなり危ない事件だったよね」と振り返った場面で展開した被害想定。

| 段階 | 起こりうる被害 |
|---|---|
| 残 0 GB | macOS が swap 確保できず apps がクラッシュ連鎖 |
| iCloud 完全停止 | Vault の変更が cloud に反映されない、複数デバイス間で divergent state |
| Voice Memos | 録音はできるが pipeline が silent fail、ideas が雲散霧消 |
| **書き込み中ファイルの破損** | Obsidian / Codex / Claude などが半端な状態で save 中断、db 破損もありうる |
| Time Machine / APFS snapshot 失敗 | バックアップが取れなくなる → 復旧手段消失 |
| macOS の自動アップデート不可 | セキュリティパッチ当たらない |

→ 記事クライマックスの「ヤバさの定量化」セクションに使える。読者の危機感を caliper で測れる素材。

## 素材 2：構造的にヤバかった 4 点

### A. サイレント runaway

24h 以上、**1 件も警告を出さずに進行**。disk pressure に気づくのは「Obsidian が空に見える」みたいな**症状が出てから**で、手遅れに近い。「気づかないまま破壊が進む」設計上のホラー。

### B. クロスカット被害

git の問題が iCloud を破壊し、Obsidian を破壊し、Voice Memos を破壊する。**犯人と被害者が遠い場所にいる**ので、原因特定が困難。実際、私（Claude）も最初「MCP のせい？モデル重い？」と全方位を疑った。**因果関係が遠い事件は調査困難**という普遍的教訓。

### C. ベンダー側のキルスイッチ不在

Codex 自身が「**親 .git 探索を止める設定は見つからなかった**」と認めてる（transcript 参照）。**ユーザー側で防御するしかない構造**になってる時点で、これは AI ツール設計の積み残し。**ベンダー責任論を提示できる根拠**。

### D. 増殖性

AI ツールが今後増えるたびに、それぞれの「便利な自動認識」が新しい地雷を踏む可能性。Codex 1 つでこれだけの被害なので、**似た系統の AI ツールを並走させる時のリスクは累積**する。**「AI ツール戦国時代」の集合的リスク**として論じられる。

## 素材 3：救いだった点（全部運要素）

事件が壊滅的にならなかった理由を整理すると、**全部運**だったことが見える：

- ユーザーが「あれ？」と気づく感性があった（昨日 40GB 空けたのに今日 5GB しかない、という違和感）
- ユーザーファイル本体は無事だった（`tmp_pack` という git 内部にしか被害が閉じてた）
- iCloud は cloud 側に master があったので redownload で復旧
- multi-AI 並走（Claude + Codex）で犯人特定までいけた

→ 「**運要素なしで被害ゼロにする防御策が必要**」という記事の主張に繋げる素材。

## 素材 4：「非エンジニアおじさん Yes ポチポチ説」の検証

ユーザーから出た重要な問題提起：

> 非エンジニアおじさんがよくわからず確認プロンプトで Yes をポチポチ押したのが原因だったら、ある意味諦め尽くし、皆さん気をつけてね、私も勉強します、で済む。
> Codex によればそもそもそれは原因じゃないので、基本ユーザー側では事前阻止は無理だったっぽいけど、Claude としても同じ見解？

### Claude の見解（Codex と同見解 + 1 点精度上げ）

**結論：Codex と同意見。事前阻止は実質不可能だった**。

ただし厳密に言えば、判断ポイントは 1 つ存在した：Codex `config.toml` が `[projects."/Users/takeomba"] trust_level = "trusted"` になっている事実から、**過去に「このフォルダを trust するか？」の workspace trust プロンプトを承認した瞬間**は存在したはず。

### でもそこで「いいえ」を選べたか？答えは NO

3 つの理由：

1. **プロンプトは "WHAT" は伝えるが "WHY/CONSEQUENCES" は伝えない**
   - 「このフォルダを信頼しますか？」と聞かれても、それが「親ディレクトリを遡って `.git` を探し、見つけたら定期 fetch を試行する」という意味だとは書いてない
   - 標準的な workspace trust UX はマルウェアからの防御を想定してて、ディスク逼迫リスクは想定してない

2. **「いいえ」を選んだら使えない圧力**
   - workspace trust は「ON にしないと機能制限される」が業界標準
   - 「とりあえず ON にしないと使えない」という心理的圧力
   - エンジニアでも何も考えずに ON にする

3. **2 つの独立イベントの掛け算で初めて発動**
   - trust ON にしただけでは無害
   - 2026-03-14 の `git init` 誤実行（こちらも 6 週間前の単発ミス、振り返って予防は不可能）と組み合わさって初めて事故化
   - **どちらか片方なら無害、両方揃って初めて爆発**する構造は予測困難

### 階層別に見た予防可能性

| レイヤー | 防げた可能性 | 現実性 |
|---|---|---|
| GUI 確認プロンプト | ほぼ NO | 実質不可能 |
| 「ホームを trust する必要が？」と疑問を持つ | YES だが | エンジニアでも気づかない |
| `git init` を誤実行しない | 一次原因の予防 | 6 週間前の単発ミス、振り返って予防不可能 |
| Codex 側が「親 .git を見つけました、これを repo として扱いますか？」と確認 | YES、これが本来あるべき | **ベンダー側の設計問題、ユーザー責任ではない** |

## 素材 5：記事の結論として推奨するトーン

「**Yes ポチポチ説**」で片付けるのは Codex を擁護しすぎ。よく見ると Codex の責任を非エンジニアに転嫁する話になっていて、実態と合わない。

書くなら「**非エンジニアユーザーが気をつけても防げなかったレベルの設計問題だった**」というトーンで攻める方が：
- 共感が得られる
- ベンダー側への建設的圧力にもなる
- 「AI ツール時代の構造的問題」として一段上の議論に持っていける

## 素材 6：自衛として今後できること（弱い対策しかない事実こそが本題）

実質的に有効な対策：

- GUI ツールで workspace trust を求められたら、**ホーム / `Documents` / `Downloads` 等の広い範囲は絶対に trust しない**（ピンポイント repo だけ）
- AI / IDE 系ツールの設定ファイル（`~/.codex/config.toml` 系）を**新規ツール導入時に 1 回だけ覗く習慣**
- 月 1 回くらい `df -h` で空き容量チェック

→ これらは「諦め尽くし、皆さん気をつけてね、私も勉強します」のニュアンスで書ける。**実質的にはこれくらいしか自衛策がない、という事実こそが、AI ツール時代の構造的問題**として記事の主軸にできる。

## 記事化する時の構成案（推奨）

3 部構成で攻めると映える：

1. **第 1 部：事件のナラティブ**（ディスク逼迫 → iCloud evict → 焦り → 調査 → 解決）
2. **第 2 部：犯人を特定するまで**（Claude の盲点 + Codex の発見 + multi-AI 価値）
3. **第 3 部：これは構造的な問題だ**（Yes ポチポチ説の検証 + ベンダー設計の積み残し + AI ツール時代の集合的リスク）

第 1-2 部は「面白読み物」、第 3 部で「読者に考えさせる」。tech 系読者にも non-tech 読者にも刺さる構成。
