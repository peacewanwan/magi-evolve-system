# 【番外編④】【意識低い系おじさんのAI活用術】言葉の意味も分からず開発を進めてた話

シリーズも回を重ねてきて、気づいたら記事の中に「GitHub」「デプロイ」「API」みたいな言葉がどんどん増えてきた。書いてる本人がそう感じてるくらいだから、読んでくれてる人はもっと「何言ってるんだろう」となってるかもしれない。

ということで、一回ここまでに出てきた用語を整理しておく。

ただ、正確な定義を書く気はない。というか書けない。ここに出てくる言葉のほとんどは、意味が分からないまま使い始めて、使いながらなんとなく掴んでいったものばかり。今でもなんとなくしか分かっていない。「素人が開発を進めるうちに、こういう理解に落ち着いた」くらいのゆるさで読んでもらえたらちょうどいいと思う。

---

## まず、使ってるツール・サービスの全体像

このシリーズで登場するツールは大きく分けて4つの役割がある。

「考える」ためのAIツール、「コードを管理する」ためのGitHub、「サイトを公開する」ためのVercel、「データを保存する」ためのSupabase。それに加えて、サイトそのものを作るためのNext.jsとnpmがある。

ここからはツールごとに「そのツールは何なのか」を書いた後、周辺で出てくる用語を並べていく。

---

## AIツール

**ChatGPT**（チャットジーピーティー）
OpenAIが作ったAIチャット。このシリーズでは「最初にアイディアをぶつける場所」として使っている。思いつきを投げて、方向性を固めるのに向いている。

**Gemini**（ジェミニ）
Googleが作ったAIチャット。このシリーズでは「別視点のツッコミ役」。ChatGPTで固まった方向性に対して「それ本当にいいの？」と聞くために投げる。セカンドオピニオン的な立ち位置。

**Claude**（クロード）
Anthropicが作ったAIチャット。このシリーズでは「まとめ＆実装担当」で、一番重要なポジションにいる。情報の整理、構造化、そして実装に落とすところまでやってくれる。

**Claude Code**（クロードコード）
ブラウザで会話するのがClaude、PCの中のファイルを直接いじってもらうのがClaude Code。ターミナルからコマンドラインで動くので、ファイルの作成や修正をそのままやってくれる。第12回で詳しく書いたけど、これが入ってから開発の進み方が明らかに変わった。

### AIツールで出てくる用語

**LLM**（エルエルエム）
Large Language Modelの略。ChatGPTやClaude、Geminiの中身にあたるもの。ものすごく雑に言うと「大量の文章を学習して、次に来る言葉を予測する仕組み」。ユーザーとしてはあまり意識しなくていいけど、「LLMには使用上限がある」みたいな文脈で出てくる。

**トークン**
LLMが文章を処理するときの単位。文字数とは微妙に違う。日本語だと1文字が1〜2トークンくらいになることが多い。課金の話で「トークン数に上限がある」と出てくるのはこれ。つまり、たくさん会話するとお金がかかる。

**プロンプト**
AIに投げる入力文のこと。「こういうの作って」とか「これ直して」とか、AIに話しかける文章全部がプロンプト。番外編①で書いたけど、最初は「プロンプトエンジニアリング」みたいな大層なことは何も考えていなかった。

**壁打ち**
本来はテニスの練習方法。壁に向かってボールを打つと返ってくるやつ。開発の文脈では「AIに考えを投げて、返ってくる反応を見ながらアイディアを固めていく」という意味で使っている。第3回で書いた「ChatGPTで壁打ち」がまさにこれ。相手がAIでも、投げて返ってくるだけで思考が整理される。

**指示書**
Claudeに実装を頼むときに書く、やってほしいことをまとめた文書。第6回で登場。「このファイルのここをこう変えて、新しくこのファイルを作って、最後にgit pushして」みたいな内容をひとまとまりにしたもの。これをClaude Codeにそのまま渡すと、書いてある通りに実装してくれる。このシリーズの開発スタイルを象徴する言葉だと思う。

## GitHub（ギットハブ）

コードを保存・管理するサービス。第9回で主役として登場した。Googleドライブのコード版みたいなものだけど、ただ保存するだけじゃなくて「いつ、何を変えたか」が全部記録される。Vercelと連携すると、ここにコードを送るだけで自動的にサイトが公開される。

### GitHubで出てくる用語

**リポジトリ**
GitHub上にあるプロジェクトの保管場所。1つのサービスにつき1つのリポジトリ、くらいの感覚。略して「リポ」と呼ぶ人もいる。最初は「フォルダみたいなもの」と思っておけば困らなかった。

**コミット**（commit）
「この変更を記録する」という操作。第9回で「意味は分かってない」と書いたやつ。ゲームのセーブに似ている。git commitと打つと、その時点での変更内容が保存される。後からいつでも「あのとき何を変えたか」が振り返れる。

**プッシュ**（push）
ローカルで記録した変更をGitHubに送る操作。コミットがセーブなら、プッシュはセーブデータをクラウドにアップロードするイメージ。git pushと打つと、自分のPCにあるコードがGitHubに反映される。第9回で「コピペしてた」と書いたのはこれ。

**ブランチ**（branch）
コードの「枝分かれ」。本体のコードに影響を与えずに、別の場所で作業するための仕組み。第9回で「ピンとこない」と書いたけど、今もそこまで深く理解しているわけではない。ただ「mainが本番で、それ以外は作業用」くらいは分かるようになった。

**マージ**（merge）
ブランチで作った変更を本体に統合する操作。Claude Codeが勝手にブランチを切って作業することがあるので、最後に「mainブランチにマージする」という手順が必要になる。これを忘れると、作業したのに本番に反映されない。やらかしたことがある。

**GitHub Actions**（ギットハブアクションズ）
GitHubが提供する自動実行の仕組み。「毎日AM4時にクローラーを動かす」みたいなことを設定しておくと、勝手にやってくれる。人間が毎朝起きてスクリプトを実行しなくていい。このサービスの裏方として地味に重要な存在。

## Vercel（ヴァーセル）

Webサイトを公開するためのホスティングサービス。第8回で主役として登場した。GitHubと連携しておくと、コードをプッシュするたびに自動でサイトが更新される。昔みたいにFTPでファイルをアップロードする必要がない。無料枠でも十分使える。

### Vercelで出てくる用語

**デプロイ**（deploy）
「作ったものをインターネットに公開する」こと。第8回で「体感で初めて分かった」と書いた言葉。ローカルで動いているだけの状態から、世界中の誰でもアクセスできる状態にする。昔と比べたらびっくりするくらい簡単になっている。

**ビルド**（build）
コードを「本番で動く形」に変換する処理。npm run buildと打つとこれが走る。開発中は多少雑でも動くけど、ビルドを通すと本番環境で問題が起きないかチェックされる。ビルドが通らないと公開できない。逆に言えば、ビルドが通ればとりあえず大丈夫。

**ドメイン**（domain）
サイトのURL。24offmap.jpがこのサービスのドメイン。お名前.comで取得して、Vercelに紐づけた。年間数千円かかるけど、独自ドメインがあるだけでサービスっぽさが一気に増す。

**DNS**（ディーエヌエス）
ドメインとサーバーを繋ぐための仕組み。「24offmap.jpにアクセスしたら、Vercelのこのサーバーに繋いでね」という指示を登録しておく場所。設定画面でAレコードとかCNAMEとか出てくるけど、Claudeに言われた通りの値を入力しただけ。正直あまり理解していない。でも動いている。

**SEO**（エスイーオー）
Search Engine Optimizationの略。Googleの検索結果で上の方に出るようにするための施策。サイトマップを送信したり、都道府県別のページを作ったりするのがこれ。まだ本格的には手をつけていないけど、長期的にはここが重要になってくるんだろうなと思っている。

## Supabase（スーパベース）

クラウド上のデータベースサービス。第10回で主役として登場した。中身はPostgreSQLというデータベースで、管理画面からテーブルを作ったり、SQL Editorからクエリを実行したりできる。最初に「怖いメール」が来たのも良い思い出。

### Supabaseで出てくる用語

**データベース / テーブル**
データベースは情報の保管庫で、テーブルはその中にある表のこと。Excelのシートに似ている。このサービスではeventsテーブルにイベント情報が入っていて、spotsテーブルにスポット情報が入っている。1つのサービスの中に複数のテーブルがある。

**SQL**（エスキューエル）
データベースを操作するための言語。「このテーブルからデータを取ってきて」とか「このデータを削除して」みたいな指示を文字で書く。第10回で「SQL Editorに貼って実行した」と書いたのがこれ。自分で書くことはほぼなくて、Claudeが書いたものをコピペしている。

**環境変数**（.env）
パスワードやAPIキーなど、コードに直接書きたくない情報を保管しておく仕組み。.envというファイルに書いておくと、コードの中から参照できる。GitHubに上げるとパスワードが全世界に公開されてしまうので、.envファイルはGitHubに上げないのがルール。APIキー（サービスを使うための合鍵みたいなもの）も全部ここに入れる。これを知らないと事故る。

## Next.js（ネクストジェイエス）とnpm（エヌピーエム）

Next.jsはWebサイトやアプリを作るためのフレームワーク。第7回で初めて触った。ページの表示やルーティングをいい感じにやってくれる。最初は「npm run devって打ったら画面が出た」くらいの理解だった。

npmはNode.jsのパッケージマネージャー。意味が分からなくても大丈夫、最初は「ターミナルで何かをインストールしたり、サイトを動かしたりするときに使うコマンド」くらいの理解で十分回る。npm run devで開発用のサイトが立ち上がり、npm run buildでちゃんと動くか確認できる。

### Next.js / npmで出てくる用語

**ターミナル**
黒い画面。コマンドを文字で打ち込んで操作する場所。Macだと最初から入っている。第7回から登場して、それ以降ずっとお世話になっている。最初は「この画面、何をしてるの？」状態だったけど、結局ここにコピペし続けるのが開発の基本動作だった。

**ローカル**
自分のパソコンの中、という意味。「ローカルで動かす」は「自分のPCの中だけで動かしてる状態」のこと。ローカルで動いてるだけではインターネットの誰にも見えない。公開するにはデプロイが必要。

**フロントエンド / バックエンド**
フロントエンドはユーザーが目にする画面のこと。バックエンドはユーザーからは見えない裏側の処理。このサービスでいうと、フロントエンドはNext.jsで作ったWebサイトの画面、バックエンドはSupabaseのデータベースやクローラーの処理にあたる。

## データ処理で出てくる用語

ここからはどれか1つのツールに限らず、データの取得や加工の場面で出てくる言葉をまとめる。

**API**（エーピーアイ）
Application Programming Interfaceの略。ものすごく雑に言うと「サービスとサービスを繋ぐ窓口」。Claude APIを使うとプログラムからClaudeに質問できるし、Google Maps APIを使うと地図をサイトに埋め込める。「APIキー」はその窓口を使うための合鍵みたいなもの。

**JSON**（ジェイソン）
データをやりとりするときのフォーマット。名前と値がセットになっていて、入れ子にもできる。クローラーがイベント情報を取得してClaude APIで整理すると、JSONの形で返ってくる。見た目は波括弧とコロンだらけで最初は読みにくいけど、慣れるとExcelより構造が分かりやすい。

**クローラー / スクレイピング**
クローラーは「Webサイトを巡回して情報を集めるプログラム」で、スクレイピングは「Webページから必要な情報を抜き出す技術」。第11回で主役として登場した。このサービスの心臓部。毎日AM4時に自動で動いて全国のイベント情報を集めてきてくれる。

**マークダウン**（Markdown）
テキストに簡単な記号を付けて見出しや太字を表現する書き方。##を付けると見出しになるし、---を書くと区切り線になる。実はこのnote記事自体がマークダウンで書かれている。エンジニアの世界ではドキュメントの標準的な書き方として定着している。

**正規化**
バラバラなデータを決まったルールで同じ形に揃えること。第11回で「バラバラのサイトの情報が、同じフォーマットで並んでいる」と感動した場面がこれ。みんカラから取った情報もイベントマニアから取った情報も、クローラーが正規化して同じテーブルに入れてくれる。

**コールドスタート**
「誰もいないから誰も来ない、誰も来ないから誰もいない」という鶏と卵の問題。オフ会メーカーのチェックイン機能がまさにこの状態で、スポットに誰もチェックインしていないと、新しい人もチェックインする動機がない。この壁をどう突破するかが、今一番の課題。
