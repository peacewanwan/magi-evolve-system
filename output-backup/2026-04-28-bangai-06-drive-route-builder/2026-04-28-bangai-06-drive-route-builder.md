# 【番外編(6)】経由地ポチポチをやめて、道は自分で決める

本編1で作ってるオフ会メーカーとは別に、もうひとつ新サービスの一応動くものが出来上がったので一旦紹介を挟んでおく。「ドライブルートビルダー」っていう、Googleマップに自然言語でルートを組ませるサービスです。

---

## 1. 出先で思いついた

いつものように週末日曜日の朝。ドライブで行った先の箱根のターンパイクスカイラウンジに居た。

そこから、以前も通ったルートで自宅に帰るために、iPhoneのGoogleマップ開いてどう帰ろうか考え始めた。あ、自分の車はディスプレイオーディオに換装していて、基本ナビも音楽もiPhoneのCarPlayを利用している。

さて、自宅からは、圏央道→小田厚→ターンパイク、と最短ルートで来たが、帰りがそれではつまらない。何度かやっているが、富士五湖方面まで降りて行って、道志みちを通ってワインディングを楽しみながら帰りたい。道はわかっているので、別にナビをセットしなくてもいけるのだが、帰着予想時間や、いく先の混雑状況を把握するためにルートをセットするのが癖になっている。

Googleマップは目的地への最短ルート（有料や高速の使用不使用も含め）を見つけるには便利だけど、このような「どういうルートで帰りたい（行きたい）」みたいに「意思」がある場合は、ルート設定が割とめんどくさい。通りたいルート上のスポットを指定して「経由地」を複数指定することでGoogleマップを誘導してルートを作らせる必要がある。

最近AIに発想の実現の可否を確認する癖がついてる脳みそにこんな事が浮かんだ。

「自然言語で『山中湖から道志みち通って』って言えば、Googleマップに通せるルート出せないんかな?」

スカイラウンジの駐車場で運転席に座ったままiPhoneのClaudeアプリを立ち上げて頭に浮かんだ事をそのまま聞いてみた。

最初はGoogle Maps Directions API で waypoints 指定して……みたいな話だったんだけど、何往復かしてるうちに方向が変わった。Claudeが途中で言ってきた。

「Google Maps URLs（ディープリンク）を使えば**完全無料**で実現できます。Routes APIすら要りません。」

つまり、自然言語をClaude APIで経由地に分解して、それをGoogleマップのURLパラメータに突っ込むだけ。要は手動で経由地をセットしてルート設定した結果Googleマップが生成するURLを、AIと仕組みで生成しちゃおうって考え。それをタップしたらGoogleマップが起動してそのままナビが始まる。自前で地図を持つ必要すらない。

「いや、それでいいじゃん」と思った。完全に無料、しかもユーザーは使い慣れたGoogleマップでナビできる。設計が劇的にシンプルになる。

こうやって、初手AIが出してきたこ難しい実装アイディアをシンプルに削ぎ落としていくのも特別編で書いた「AIとの付き合い方」を深めていくうちに身につくテクニックだと思う。

とはいえ、本当にできるのか？どの程度使い物になるのか？

出先なので、Claude Codeに試験的にCode実装させるわけにもいかない・・・・・

どうしたものか・・・いや、まあアイディアとして保存して後でゆっくり設計すればいいんだけど、AIと会話しながら何かを作るっていうのを日常的にやっていると、ここまでアイディアが具体化してくるともう一歩進めたくなる。で思いついたのが

「アプリ化する前に、いまここで目的地と帰りたいルート言ったらurl出せる?」

Claudeが「できます」と返してきたので、プロンプトからURL作成っていう方法で試してみた。これがちゃんと機能すればWebサービス化は簡単（実際やるのはAIだけどw）

```
出発地 ターンパイクスカイラウンジ
目的地 多摩センター
希望ルート
山中湖方面おりて、道志みちを通って道の駅どうし寄ってから目的地まで
```

Claudeから生成されたURLが返ってきた。タップした。Googleマップが立ち上がって、ターンパイクスカイラウンジから山中湖、道の駅どうしを経由して、多摩センター駅までのルートが表示された。なぜ多摩センターか？自宅の住所打つのが面倒だったので自宅に近いランドマーク的スポットにしただけw

思った通りのルートだった。

> おーすげー
> 思った通りのルートが出た!

これが朝8時06分。スカイラウンジ駐車場でiPhoneいじってるおじさんが、自分一人でテンション上がってた。

起点は壮大な構想じゃない。「現地で困った」「Claudeに聞いたら解決した」、それだけ。

## 2. 4日で動いた

帰宅して、Claude Codeにこの話を渡した。渡したっていうか、スカイラウンジでひと段落した後にClaudeアプリに書かせた引き継ぎ用のMDファイルをダウンロードしてClaude Codeに読ませただけ。そこから4日でテスト版が動いた。

ざっくり時系列。

- 1日目: 命名プロンプトを作って、初期辞書（道志みちとかターンパイクとかの主要ドライブルート）を準備、テスト版実装に着手
- 翌日: 修正4本立て続け。waypoints重複除去、temperature=0で再現性確保、avoid_paramsの調整、LP active 化
- その翌々日: Phase 1.5 を一気に流し込み。共有4ボタン、マイプレイス機能、使い方ページ、移動手段プルダウン

ドライブ先で朝思いついてから、Phase 1.5までが4日間。中身濃い目に動いて1週間ちょい。

（上記中身はAIに整理してもらったのを並べただけ。特に2日目の内容なんて何やってるか全然わかってない）

なぜそんなに速いか。すごいからじゃない。土台があったから。

本編からずっと作ってきたWebサービスで、Next.js + Supabase + Claude API + Google Maps API の組み合わせには既になんとなく理解してた。デザインシステムも、デプロイ環境も、認証周りも、共有ボタンの作法も、全部一回通った道。

だから新サービスといっても、ゼロから組み立てたんじゃない。同じ部品を組み替えただけ。ここまで積んできた疲労と試行錯誤の上に、ぽんと乗っかった感じ。

・・・みたいに書くとなんかすごく「成長しながら作ってる」風だけど全然違う。相変わらず仕組みや構造、技術的内容はざっくりとしか理解してなくて「あれができるんだからもしかしてこれもできる？」程度の理解度。

思いつきから実装までの速さは、何が出来て何は難しくて、どう聞けば、指示すれば自分の考えに近しい提案・答えを導き出せるか？等のAIとの対話作法がある程度自分の中で出来上がってきているからだと思う。

対話しながら仕様さえ固まってしまえば実装はものの数分で完了する。

## 3. テストと本格実装

テスト版が動いたところで知り合い何人かに渡してみた。「ドライブ計画立てるのが面倒なときに試してみて」と。

特に大きな不具合はないっぽい。ただバイク乗りの友人からは「自動車専用道路避けられないの？」と言われた。これに関しては後述。

とりあえず使えるのはわかったので、構想にある機能を順次実装していった。

まず**ルート再調整**。出てきたルートが思ったのと違ったら、追加指示で再生成できる仕組み。「もっと海沿いで」「○○PA経由で」と言い直すと、別ルートが返ってくる。同じセッション内で何回でも調整できる。

次に**SNS共有ボタン**を追加。X・Threads・Instagram・コピー。ワンタップで投稿フォーマット込みで飛べるように。Instagramだけは公開APIないのでURLコピー＋案内表示で妥協。

そのあと**マイプレイス機能**。自宅とか友人宅、愛人宅とか、週末よくいくスポットとか、よく使う場所を最大10件登録できる。すべて localStorage 保存で、サーバーには一切送らない。これは後のプライバシー設計の話とつながる。

**使い方解説ページ**（`/drive/how-to-use`）を別建てで追加。「なぜ作ったのか」「自然言語入力のコツ」「マイプレイスの説明」「プライバシー方針」を5セクションで。サービス本体に説明文ベタ貼りはUIが重くなるから分離した。使ってみる時はまず最初に一読して欲しい。

最後に**移動手段選択（車・自転車・徒歩）**。

ここまでで Phase 1.5 が一通り出揃った。書き出すと結構いろいろ足してるように見えるけど、実装はAIだから1機能あたりやり取り込みで数十分。自分は「これ足したい」「これはやっぱ要らなかった」って判断するだけ。

機能を足していった結果として、UIは何度か組み直しになった。手戻りも何回かあった。出して外して、また出して、みたいな。所詮個人開発のプロダクトなので利用者とか気にせず、タイミングも測らずにどんどん更新していった。

## 4. 二輪モードの罠

上で書いた移動手段「車・自転車・徒歩」の3択。

最初は友人からのリクエストもあったので「バイク」も追加して4択にしたかった。でも、Googleマップの通常UIにあるのは、車、電車（公共交通）、徒歩、自転車、飛行機だけ。

でもとりあえず、確認してみようか。って事でAIに調査を依頼。どうやらGoogleマップのURL APIドキュメントに `travelmode=two-wheeler` ってのがあるらしい。ただ、利用の可否が国ごとになっていて、日本では未開放らしい。Claudeは「やってみないとわからない」という。

念の為、Geminiに確認したら「日本は未対応」とのこと。

ダメか・・・・

とはいえ実装コスト・時間はほぼゼロなのでClaude Codeにお願いして実装して、検証で叩いた。

エラーが返ってきた。

> このルートは現在のところ、オートバイ ルート検索の対象圏外のようです

やっぱりダメだった。こればっかりはしょうがない。

UIからは外した。でも `travelMode` の type 定義には `two-wheeler` を残してある。インド進出するときには使える(しない)。あるいは Google が日本対応してくれた瞬間に1行コメントアウト外せば有効化できる。

地味だけど、こういう「一回ハマったやつのコードコメント」が後で効く。半年後の自分は絶対忘れる。

## 5. プライバシー設計

同じく上記で機能として追加した「ルート再調整」。最初に出てきたルートが想定と違ったら、追加指示を入れて再生成できる、というやつ。

このとき、フィードバックログを溜めるテーブルを作った。`route_feedback` という名前。あとで見返して「どういう追加指示が多いか」を分析して、プロンプトを育てる用途。

ただし、保存しないものを明示的に決めた。**出発地と目的地は保存しない。**

ログに残すのは、自然言語の入力テキストと、抽出された経由地の配列、追加指示の内容、これくらい。出発地と目的地そのものは記録外。

理由としては単純で、出発地や目的地は自宅や、愛人宅等プライベートな場所が指定される可能性が高いためデータとして保管するのはリスクが高いと思ったから。「ターンパイクスカイラウンジ → 自宅」みたいなログが溜まったら、ユーザーごとの自宅座標が筒抜けになる。仮に自分がユーザーだったら、絶対嫌だ。

AIに任せて作ると、こういうとこ意外と作り込んじゃう。「ログは全部取っておいたほうが分析できる」みたいな素直な発想で、丸ごと保存する設計が出てくる。一回出してきた設計を「いや、ここは取らないで」と外すのは、自分が判断するしかない。

「俺だったら絶対嫌」基準は、この手のサービスを作るときの最終チェックとしてわかりやすい。データを取るかどうかの判断で迷ったら、これに戻る。

エンジニアじゃない自分は、システム作りの基本的作法とか絶対の掟とかはよく知らない。ただ「自分がユーザーだったらどう思うか」を判断基準として注意するようにしてる。

## 6. 命名「ドライブルートビルダー」

開発中はずっと「ルートメーカー」という仮名で呼んでた。もう１つのサービスに「オフ会メーカー」というのがあるので、語感を揃えた感じ。

ただある程度機能が揃った段階で、サービス名を確定させたくなった。このままルートメーカーで進むとタイトルやらノートの記事やら後から変えると色々不都合が出てくる。

ここで、本編⑰でやったのと同じパターンを使った。**3つのLLMに命名プロンプトを投げて合議させる。** ChatGPT・Gemini・Claude.ai の3つに、サービスの目的・機能・ターゲットを同じプロンプトで渡して、それぞれに命名候補を出してもらう。本編⑰のサイト名命名と同じやり方。

<!-- 本編⑰公開後にここへリンク追記予定 -->

結果、「ドライブルート」を軸にする方向は3者一致。その上で「メーカー」「プランナー」「ジェネレーター」「ビルダー」あたりが候補に挙がった。

最終的に「ドライブルートビルダー」に決めた。

決め手は「ビルダー」のニュアンス。「メーカー」だと自動生成感が強くて、「自分はボタン押すだけ」感が出る。「ビルダー」は「ユーザーが自分で組み立てる」感じが入る。実際このサービスは、自然言語で経由地を指示するところがコアなので、ユーザーの意思が入る言葉のほうが合う。

オフ会メーカーとは語感が揃わなくなった。でもサービスの性質が違うんだから、無理に揃えなくていい。そう判断した。

## 7. 試してみてほしい

ドライブルートビルダー、公開済み。

URL: https://24offmap.jp/drive

特徴3点。

- **自然言語1発でルート生成**: 「山中湖経由で道志みち通って」みたいに書けばOK。経由地ポチポチは不要
- **マイプレイス**: 自宅とかガレージとかを最大10件、localStorageに保存可能。**サーバーには一切送らない**
- **ルート再調整**: 最初に出たルートが違ったら、追加指示で再生成。「もっと海沿いで」とか「○○PA経由で」とか

経由地ポチポチが面倒だと感じたことがある人、試してみてほしい。バグ報告・「こういう機能ほしい」も歓迎。

特に、自分の知らない名道（峠・スカイライン・林道）でうまくルートが出ない場合は教えてほしい。Claudeに食わせる主要ドライブルート辞書がまだ薄い。フィードバックもらえるとそのまま改善材料になる。

なお冒頭で書いた「ターンパイクスカイラウンジ → 山中湖 → 道の駅どうし → 多摩センター駅」のルートは今でも同じ入力でちゃんと動く。あの朝のテストURLが、サービスとしてそのまま手元に残った形になった。これがちょっと、嬉しい。
