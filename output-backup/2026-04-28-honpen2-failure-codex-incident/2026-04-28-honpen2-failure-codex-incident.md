# 【AI失敗パターン編】AIが暴走したっぽいのでAIに捜査させたら犯人が特定できた

## 3日前のニュース、笑えなかった

数日前、Xで流れてきたこんな記事を見た。Tom's Hardwareの記事で、Cursor上のClaudeがPocketOSという会社の本番DBを9秒で消した。ステージングのつもりが本番のボリュームを巻き込んでた、という話。

https://www.tomshardware.com/tech-industry/artificial-intelligence/claude-powered-ai-coding-agent-deletes-entire-company-database-in-9-seconds-backups-zapped-after-cursor-tool-powered-by-anthropics-claude-goes-rogue

普通なら「うわ、こわ」で終わる。でも自分は笑えなかった。つい最近自分のMacBookもAIにやられてたから。

向こうは本番DB全消し、こっちは64GB分のディスクを食われただけだから規模は全然違う。ただ、構造は近い。AIが裏で勝手に判断して、ユーザーが知らないうちに環境を壊す。それが、わりと普通に？起きる可能性があるみたいだ。

今日はその、自分が食らった方の話を書いておこうと思う。

## ファイルが見えない

2026年4月某日、開発やnote執筆に使ってるMacBook Air（M3、SSD 228GB）の空き容量が急に減った。というか、出先からから帰ってきて作業をしようと思ったら、iCloudを参照フォルダにしているObsidian（自分が使ってるメモ・ナレッジ管理ツール）のデータが見れない。指定保管庫（Obsidianでデータを置いてるフォルダのこと）以下のフォルダが全て見られない。
かなり慌てた。Macの環境設定やら、Obsidianの設定やら色々調べて、どこでだか忘れたけどこんな文言を見つけた「Diskの空き容量が無いためiCloudの同期を中止しました」。

へ？？
たまたま前日にディスクの空き容量が逼迫したため、いらないファイルや使ってないアプリを削除して空き容量を増やして40GBほどの空き容量を増やしたばっかりだから、？？？？ってなった。何事？
メインHDD（250GB SSD）の状態調べたら空き容量 200MBちょい
なんだこれ？

ちなみにMacBookは開きっぱなしなのでサスペンド等には入らずずっと起動しっぱなしの状態だった。わけがわからないのでとりあえずもう習慣化していAIに聞く、をやってみた。

「昨日空き容量増やしたはずなんだけど、空きが全然なくなってる。原因調べられる？」
相変わらずプロンプトエンジニアリング感ゼロの尋ね方

そこで出てきたのは・・・・・覚えてないしそもそも理解できなかったからAIに書いてもらう

---

頼まれたので、こちらで起きていたことを整理します。

調査したところ、ホームディレクトリ直下に `~/.git` というGitリポジトリが存在していました。本来そこに置かれるべきものではありません。中の `objects/pack/` フォルダに、917MBの `tmp_pack_*` ファイルが大量に並んでいて、1時間ごとに1つずつ増えています。24時間以上ずっと続いていて、合計で約64GBに達していたのが容量逼迫の正体です。

念のため `~/.git/config` の内容を確認したところ、`remote = https://github.com/xxxxxxx/xxx-xxxxx-xxx.git` と記載されていました。アンダースコア付きの記述で、現在運用されているリポジトリ `xxx-xxxxx-xxx`（ハイフン）とは別のものです。

経緯を辿ると、6週間前に、ホームで `git init` が誤って実行された残骸であることが分かりました。打った犯人はご本人です。

ただ、6週間放置されていた `.git` が、なぜつい最近になってネットワーク経由でデータを取得し始めたのか、この時点ではまだ特定できていません。原因が掴めないと、再発防止の対策も決められない状況です。

---

ということらしい。原因はわかった（らしい）。直せるの？って聞いたら安全な直し方も調査してくれてとりあえず空き領域が60GB超まで回復した。100％とは言えないが打てる対策も打ってくれた（らしい）

## おまえか・・・

Codexというのは、Claudeとは別のAIコーディングエージェント。最近導入して Claude と並行で動かしてる。並行というか、Claudeを制作担当とすればこちらは品質保証担当として数日前に導入して、Claude Codeとの情報共有の仕組みも構築済み。自分がClaude Codeで何をやっているか、何を作っているかも把握済みの状態だ。

早速Claudeが作ったプロンプトを投げる。


数分間Codexは色々調査して戻ってきた。後で調査ログを見たClaudeによると`~/.codex/config.toml`、`state_5.sqlite`、`logs_2.sqlite` を全部漁ってたらしい

ログを確認したClaudeにどういう事だったか、説明してもらおう

---

代わりに整理します。

Codexが自分の `~/.codex/config.toml` を確認したところ、以下のような記述が残っていました。

```toml
[projects."/Users/xxxxxxxx"]
trust_level = "trusted"
```

ホームディレクトリ直下が trusted project として登録されていた、ということです。さらに `state_5.sqlite` の thread metadata に、2026年4月24日の15時58分と22時32分の2件、`cwd=/Users/xxxxxx/xxxxx` で `git_origin_url=...xxx-xxxxxx-xxxx.git` というレコードが残っていました。

この時刻が、tmp_pack の累積が始まった時刻と一致しています。Codexが xxxxxxを開いた瞬間に、親ディレクトリの `~/.git` を repo として拾い、自動 fetch を回し始めた、という構図のようです。

そのあと、自分のバックアップを取って、ホーム直下の trust block（さっきの「信頼するプロジェクト」登録部分）だけ静かに削除して、報告書を出してきた。


ここで一旦、立ち止まる必要がある。今書いた「Codexが原因だった」という結論は、Codex自身が自分のDBを漁って「自分が原因」と言った内容。それを人間側で独立に検証する手段は、自分の様な非エンジニアにはない。AIの中で起きてることは、AIが説明してくれてる事を信じる事しかできない。それしか選択肢がなかった、と書いた方が正確かもしれない。
一応Codexの自白？をClaudeに渡して確認してもらうことはしたが・・・結局せかんどおぴにおんもAI頼り。

## 結局、AIで診てAIを信じた

整理するとこうなる。

AIが起こしたかもしれない問題を、別のAIに診断させて、そのAIが出した「自分が原因です」という説明を受け取って、AIが対策を打った。全工程にAIが絡んでて、人間がというか自分が独自に検証する余地は、ほぼなかった。

それでも結果として、64GBは回収できた。一応二度と同じ様なことが発生しない対策もしてくれたらしい。

これで本当に直ってるのかも、厳密には分からない。AIがまた別の種類の暴走をしたら同じ様な、いやもっと深刻なトラブルが起きる可能性はある。

ここに今書いてる「Codexが原因だった」という記述も、突き詰めれば「CodexとClaudeがそう言ったから自分はそう書いている」でしかない。

別AIに診断させる構造が機能した、と前向きに読むこともできる。AIのことはAIに聞くしかなかった、と消極的に読むこともできる。同じ事実が、見方によって違って見える。

## 付き合い方は人次第

今回打った対策はこんな感じ、、というのをAIに教えてもらった。
どんな内容でどんな効果があるのか自分ではよくわかってない。

- `~/.git` の場所に sentinel ファイル設置（物理的に書き込みブロック）
- Codexの trusted project から `/Users/xxxxxxxx` を削除
- `GIT_CEILING_DIRECTORIES=/Users/xxxxxxxxx` の導入を検討中

ただ、対策を打ったところで、何度も言うがAIが裏で何をしてるかは結局分からない。これも繰り返しになるけど、これからも別の形で似た事故が起きる可能性はある。

それを理解した上で、AIとの付き合い方は人によって違っていいと思う。
こういう想定外のリスクが有る事を理解した上でAIを信じてAIと付き合うか、リスクを回避してAIとの付き合い方を限定的にするか、それは使う人の考え方次第


自分はAでいく。所詮個人のパソコンで個人で開発しているので、最悪でもMacBookをクリーンにして構築し直せばなんとかなるでしょ。とはいえ、外部にサービスを公開してもいるので、これを機会にAIにリスクアセスメントをさせようとは思う。

ただ、Bが間違ってるとも思わないし、Cが時代遅れだとも思わない。「9秒で企業の本番DBが消える」よりはおそらく身近に感じてもらえる自分の体験を参考に残しておく。
